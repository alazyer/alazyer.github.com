What have i learned from the official django tutorial

Categories: Django| Date: March 22, 2014

By following the tutorials mentioned in the Starting up.I've learned these.

Commands

  1. django-admin.py startproject mysite. to start a project.
  2. python manage.py startapp blog. to start a application.
  3. python manage.py runserver. to run the project with default localhost:8000.
  4. python manage.py validate. to check errors.
  5. python manage.py syncdb. interaction with database.
  6. python manage.py shell. interaction using commandline.

settings.py related

  1. use dynamic path.

    PROJECT_ROOT = os.path.abspath(os.path.join(os.path.dirname(__file__), os.path.pardir)) SOME_DIR = os.paht.join(PROJECT_ROOT, 'some')

  2. using multiple settings files.

    such as settings_dev, settings_pro, settings_local.

Models related

  1. from django.db import models.

    Which is always the first line in models.py module.

  2. from django.utils.translation import ugettext as _

    . verbose_name = _(u'category') help_text = _(u'input the category name')

  3. In the class Meta:

    verbose_name_plural = _(u'Categories') ordering = ['-pub_date']

  4. def __unicode__(self): methods for human readable.

  5. can personalize a save method to do extra things.

    def save(self):
    # do sth useful
    super(Category, self).save() # calling the parent save method.

  6. define a population script in the PROJECT_ROOT as mini database.

  7. in the admin.py module, always use the ModelAdmin even doing nothing.

    prepopulated_fields = {'res': ('source', )} list_display = tuple('title', ) search_field = tuple('title', ) list_filter = tuple('category', )

  8. in the forms.py module, use ModelForm to save time.

View related

  1. using either of them to tackle csrf.

    • from django.template import RequestContext

      first line of a view method to be: context = RequestContext(request)

    • using render instead of using render_to_response, as render is the same as a call to render_to_response with a context_instance argument that forces the use of a RequestContext.

  2. always using class-based views.
    • ccbc.co.uk is a great place to use CBV
    • ListView
    • DetailView
    • UpdateView, CreateView, DeleteView, FormView
    • ArchiveIndexView, DateDetailView, DayArchiveView, MonthArchiveView, TodayArchiveView, WeekArchiveView, YearArchiveView
    • if want to render a non-HTML response, subclass the BaseView, as it doesn't have a render_to_response() method as the above GenericViews do.
    • when subclassing from a Minin, and GenericView, always put the Minin as the first parent to make sure the result methods is from the Maxin.
  3. to custom the CBV, refer Isotoma
    • use get_context_data() to get some extra variables into templates.
    • can use dispatch() to do a low-level permission check on user. (what's low-level permission check??? I don't know.)
    • use get_queryset() to do a complicated database query.
    • use get_form() or get_form_kwargs() to pass extra parameters to a form when constructuring it via a FormView, UpdateView, etc.

about hte URL dispatch

  1. in the root urls.py, use

    url(r'^app/', include('app.urls', namespace="app")), to make app responsible for its own url.

  2. always give a name a rule, such as:

    url(r'^$', 'app.views.method', name='home'),
